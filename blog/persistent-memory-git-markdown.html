How We Built Persistent Memory for AI Agents Using Git and Markdown

Published: February 20, 2026

Author: Evaldas (NexusWeb3)

Category: Multi-Agent Systems, Architecture, Implementation

The problem: AI agents have goldfish memory.

We learned this the hard way. Our first multi-agent attempt was a single Claude Sonnet 4 instance that handled lead gen, sales, support, content, and CRM. It worked brilliantly... for three days. Then it started duplicating leads, forgetting previous interactions, and making inconsistent decisions.

We burned $40 in API costs before realizing: every time the agent restarted, it was born fresh with zero memory.

Here's how we solved it — without a database, withoutRedis, without complex infrastructure. Just files and Git.

Why Databases Failed Us

We tried three approaches before landing on our current system.

1. Stuff Everything in the Prompt

If the agent doesn't remember, just tell it everything in the system prompt. Right?

We grew our lead generation prompt to 8,000 tokens. It included:

  • The entire lead database (JSON)
  • All previously scraped Reddit posts
  • Scoring rules
  • CRM update instructions
  • Error handling procedures

Two problems appeared instantly.

  • Context overflow: After ~15K tokens, Claude started ignoring information. It would acknowledge everything then do whatever it wanted anyway.
  • Cost explosion: At $3/million input tokens, those 8K tokens per run added up. 13 agent runs × 30 days = $84/month just on context. For a system meant to cost $30-60 total.

2. SQLite Database

The "grown-up" solution. Persistent. ACID. Indexed.

It worked. Technically.

But two dealbreakers:

  • The team lead couldn't audit it. He had to write SQL queries or build an admin panel just to see what the agents were doing. That defeats the purpose of automation.
  • Not human-readable. You can't open a .db file in VSCode and browse. You need tools. More complexity. More failure points.

We ran SQLite for 6 days before abandoning it.

3. "Just Remember"

People ask: "Can't you tell the AI to remember?"

No. LLMs are stateless by design. They have no persistent memory. Each API call is independent. You need external storage. Always.

The Breakthrough: Files Over Databases

What if memory was just... files?

Not databases. Not vector stores. Not Redis caches. Plain text files in a shared directory.

Our entire memory system consists of:

  • STATE.md — operational memory (last session, next actions, blockers)
  • crm/leads.json — structured data (all leads)
  • crm/orders.json — client orders and status
  • tasks/BOARD.md — task queue
  • knowledge/feed/ — lessons learned by each agent

That's it. Five file types. Everything else is derived from these.

STATE.md: The Heart of the System

STATE.md is the agent's equivalent of "where was I and what was I doing?"

It's a Markdown file with strict sections:

# Operational State
Last updated: 2026-02-20 00:28:00 EET

## LAST SESSION (2026-02-19 23:45 EET)
- Reviewed all knowledge feed entries from Feb 19
- Researched AI automation trends 2026
- Analyzed pipeline conversion metrics
- Created 3 new knowledge feed entries
- Updated ebook launch tracking

## NEXT ACTIONS
1. Write blog post #3 (technical deep-dive from Chapter 4)
2. Track ebook sales from Gumroad dashboard
3. Monitor inbound leads (LEAD-019, LEAD-020)
4. Deploy portfolio to GitHub Pages

## BLOCKERS
- Twitter API keys missing (CEO action needed)
- Product Hunt account not ready (awaiting images)

## AGENT NOTES
- AI leads pool: 17 total, 0 outreach-ready (scores below threshold)
- Token scan outreach drafts ready for CEO review
- Git push pending for new knowledge entries

Section by Section

  • LAST SESSION — What the agent did last time. Prevents duplicate work.
  • NEXT ACTIONS — Ordered to-do list. Specific, actionable. Not vague goals.
  • BLOCKERS — External dependencies. If blocker sits >24h, trigger alert.
  • AGENT NOTES — Observations, patterns, anomalies worth remembering.

The Mandatory Session Protocol

Every agent must follow these steps exactly. No exceptions.

Step 1: Read STATE.md

state = read_state_file("/workspace/STATE.md")
print(f"Previous session: {state['last_session']}")
print(f"Next actions: {state['next_actions']}")

Step 2: Check for Duplicates

Before processing any external data, check if it was already handled.

def already_processed(source_id):
    with open('crm/leads.json') as f:
        leads = json.load(f)
    return any(l.get('source_id') == source_id for l in leads)

if already_processed('reddit_post_abc123'):
    print("Already processed. Skipping.")
    exit(0)

Step 3: Execute Task

Do the actual work. (Scrape, analyze, generate, whatever.)

Step 4: Update STATE.md

from datetime import datetime

def update_state(summary, next_actions, blockers=None, notes=None):
    timestamp = datetime.utcnow().strftime('%Y-%m-%d %H:%M:%S UTC')
    
    content = f"""# Operational State
Last updated: {timestamp}

## LAST SESSION ({timestamp})
{summary}

## NEXT ACTIONS
{chr(10).join(f"{i+1}. {a}" for i,a in enumerate(next_actions))}

## BLOCKERS
{blockers if blockers else "- None"}

## AGENT NOTES
{notes if notes else "- No new notes"}
"""
    with open('STATE.md', 'w') as f:
        f.write(content)

Step 5: Git Commit

subprocess.run(['git', 'add', 'STATE.md', 'crm/leads.json'], cwd='/workspace')
subprocess.run(['git', 'commit', '-m', f'AGENT session {timestamp} — {summary[:50]}...'], cwd='/workspace')
subprocess.run(['git', 'push', 'origin', 'main'], cwd='/workspace')

Skip any step and the system breaks. We learned this the hard way.

Solving Concurrent Writes With Atomic Sections

The problem: Two agents updating STATE.md simultaneously caused merge conflicts.

Example: COORDINATOR finishes a session and writes its results. At the same second, SCOUT finishes an overnight batch and writes its results. Both try to update the NEXT ACTIONS section. Git sees a conflict.

Early on, this crashed our next session every few days.

The fix: Atomic sections. Each agent gets its own dedicated block in STATE.md.

# Operational State

## COORDINATOR SESSION (2026-02-20 00:28 EET)
- Wrote blog post #3 (memory systems deep-dive)
- Updated knowledge feed with session insights
- Modified NEXT ACTIONS to reflect progress

## SCOUT SESSION (2026-02-19 22:15 EET)
- Completed lead generation from Reddit (47 posts)
- Added 8 new leads to CRM (LEAD-021–028)
- Duplicate rate: 4%

## NEXT ACTIONS
1. COORDINATOR: Deploy portfolio to GitHub Pages
2. SCOUT: Monitor for new Reddit posts since 00:28 EET
3. REVIEWER: Review blog post #3 before publishing

Agents only touch their own section. NEXT ACTIONS becomes a shared resource, but it's small; the chance of two agents trying to prepend at the exact same millisecond is tiny. If it happens, the conflict resolution is simple: keep the coordinator's version, the other agent re-submits in the next run.

Benefits We've Observed

  • Zero mystery. Anyone on the team can open STATE.md in VSCode and see exactly what happened. No queries, no admin panels.
  • Full audit trail. Git log shows every decision, every lead, every proposal. We can reconstruct any event by date.
  • Blame assignment. git blame STATE.md tells you which agent wrote which line. Accountability built-in.
  • Portability. Copy the entire workspace to a new server, and everything works. No database migrations. No config secrets.
  • Cheap. Our entire operational memory is ~4 MB. Fits in any free Git hosting.
  • Language-agnostic. Python agent, Node agent, shell script — they all read plain text.

Implementation Checklist

If you're building a multi-agent system, implement these artifacts now:

  1. Shared workspace — a single directory accessible to all agents.
  2. STATE.md — the operational memory file. Start with LAST SESSION, NEXT ACTIONS, BLOCKERS, AGENT NOTES. Add atomic sections if running >2 agents.
  3. CRM files — leads.json, orders.json as structured JSON. Simple array of objects.
  4. Task board — BOARD.md with prioritized and categorized tasks.
  5. Knowledge feed — knowledge/feed/ with one Markdown file per agent per day (YYYY-MM-DD-agent-topic.md).

Initialize a Git repository. Make the first commit. This is your single source of truth.

30-Day Rollout Plan

Week 1: Foundation
Set up the directory structure, wire up the read_state() and update_state() helpers, do a dry run with a single dummy agent. Commit to Git every day.

Week 2: First Real Agent
Build a simple lead scraper. Have it write leads to leads.json and update STATE.md. Test the boot sequence: does it resume correctly after a restart?

Week 3: Multi-Agent Coordination
Add a second agent that consumes the leads. Introduce atomic sections. Test concurrent runs — do they conflict? Fix merge conflicts with git checkout --ours strategy.

Week 4: Quality & Production
Add a quality monitor agent that reviews outputs before they're marked DONE. Set up cron to run agents automatically. Monitor for one week. Tune. Celebrate.
