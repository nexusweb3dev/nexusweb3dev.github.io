Why Most Agent Systems Fail

Build a simple agent. It works. Then you add a second. They start stepping on each other's toes.

Common failure modes:

  • Shared state collisions — Two agents write to the same file simultaneously
  • Cascading failures — One agent crashes, pulls down the whole workflow
  • Runaway costs — Unbounded context growth, redundant model calls
  • Context loss on restart — No persistence, agents forget everything
  • Orchestration bottlenecks — Everything goes through a single manager

We've hit all of these. Here's how we solved them.

Not every task needs GPT-4o. We route:

Implementation: Simple Python router based on agent profile.

Pattern 2: Persistent Memory via Git + Markdown

We tried databases. Too complex. Markdown files in a Git repo:

  • Each agent has its own memory/ folder
  • All writes are atomic commits (file lock prevents collisions)
  • History is automatically versioned
  • Easy to inspect, diff, rollback
with git_lock('memory/lead_qualifier.md'):
    append_memory('processed_leads', lead_id)
    git_commit('Lead qualifier update')

Downside: Git operations add ~200ms latency. Acceptable for our throughput (few writes/hour).

Pattern 3: Graceful Degradation

When an agent fails, the system doesn't crash.

Retry policy:

  1. Immediate retry (same model, same prompt) — transient errors
  2. Fallback to cheaper model — quality acceptable
  3. Alert to human via Telegram

Example: Image generator fails → retry 2x → fallback to text-only plan → notify.

No single point of failure.

Pattern 4: Cost Governance

We enforce:

  • Daily budget per agent (prevents runaway loops)
  • Context token limit (hard cap at 8K tokens for routine work)
  • Model whitelist (no surprise $15 calls)
  • Usage logging to costs/ CSV for weekly review

Our bill stayed under $50/month for 9 agents running continuously.

Pattern 5: Human-in-the-Loop Checkpoints

Fully autonomous is dangerous. We insert human approvals at:

  • Proposal send (CEO reviews Reddit DMs)
  • Blog publication (SENTINEL QA, NEXUS deploy)
  • Token scan reports (manual code review before delivery)

Everything else runs hands-off.

Trade-offs and Limitations

  • Not real-time — Agents run on schedules (minute to hourly)
  • No streaming — Batch processing only
  • Simple orchestration — Cron-driven, no DAG
  • Language-bound — All Claude Code; no cross-model agents

Works for us because our tasks are asynchronous: content production, lead processing, reporting.
