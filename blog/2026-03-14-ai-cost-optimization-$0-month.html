Pattern 1: Model Routing by Task Criticality

The principle: Use the cheapest model that gets the job done.

Not every task needs GPT-4. In fact, very few do.

We categorize tasks into three tiers:

High-Stakes (Premium Models—Use Sparingly)

Tasks where errors are costly or outputs are client-facing. These may require paid models when free tiers hit limits:

  • Project proposals: Claude Sonnet 4 (limited free tier, fallback to GPT-4o-mini)
  • Technical analysis: GPT-4o (fallback to GPT-4o-mini)
  • Client deliverables: Premium models only when explicitly paid by client

Rationale: These are revenue-critical. Use free tiers first; only upgrade if quality insufficient.

Medium-Stakes (Free Tier Workhorses)

Tasks that matter but have tolerance for minor issues—perfect for free tier:

  • Lead scoring: StepFun free module or GPT-4o-mini
  • Content drafts: StepFun free module or Claude Haiku free tier
  • Internal reports: Any free tier model

Rationale: Free tier models are 90% as good as paid for internal tasks. The 10% difference is acceptable.

Low-Stakes (Absolutely Free)

Routine processing, data transformation, simple classification:

  • CRM validation: StepFun free module
  • File formatting: Local scripts (no LLM needed)
  • Image generation: Pollinations.ai (free)

Rationale: These tasks are commodity. Zero cost is possible.

The Compression Techniques

1. Summarize, don't archive
Instead of including the full lead history (2,000 tokens), we summarize key signals:

# Lead Summary (for scoring)
- Platform: Reddit
- Budget signals: "$3-5K/mo", "500/day"
- Urgency: "ASAP"
- Technical depth: "Need AI chatbot"
- Score history: prev=45, now=60 (improved)

That's ~150 tokens vs 2,000+.

2. Selective inclusion
When writing content, we don't send the entire knowledge base. We send:

  • The specific topic (2 lines)
  • Target audience (1 line)
  • Tone guidelines (3 lines)
  • Previous similar posts for reference (3 excerpts, 100 tokens each)

Total: ~500 tokens vs. sending 5,000 tokens.

3. Reference external files, don't embed
Static reference data lives in separate files. Agents read them on demand, not pushed into every prompt.
If an agent needs "pricing," it reads pricing_table.json — a separate operation we cache.

Pattern 3: Async + Batching Over Real-Time

Real-time hits rate limits faster.

Free tiers have requests-per-minute limits. Batching maximizes throughput.

Why Batching Saves Rate Limit Budget

Processing one-by-one uses API calls inefficiently:

  • Each call: ~50–200 tokens of system prompt overhead
  • Over 100 leads: 5,000–20,000 extra tokens/day = wasted RPD quota

Batch 100 leads into a single call:

  • One system prompt (200 tokens)
  • Total overhead: 200 tokens vs 5,000–20,000

Rate limit savings: 96–99% — you can process more items within free tier quotas.

Pattern 5: Caching + Memoization

Don't recompute what you already computed.

We cache LLM responses for identical inputs. This conserves rate limit quota and speeds up execution.

Example: Lead Enrichment

  • Lead #1: company=acme.com → fetch (1 API call) → cache
  • Lead #2: company=acme.com → cache hit (0 API calls)
  • Lead #10,000 with same company? Still cache hit.

Impact: For recurring leads, we save 90% of API calls, preserving free tier quota for new work.

Implementation

response_cache = {}

def cached_llm_call(prompt: str, model: str):
    cache_key = hash(prompt + model)
    if cache_key in response_cache:
        return response_cache[cache_key]
    response = llm.ask(prompt, model=model)
    response_cache[cache_key] = response
    return response

Cache persists to disk between runs. Cache hit rate: ~35%. This effectively multiplies our free tier quota.

Pattern 6: Human-in-the-Loop for High-Stakes Only

The most underrated optimization: don't automate everything.

We let agents run autonomously for 90% of tasks. The remaining 10% require human review.

What Gets Human Review

  • Proposals sent to leads scoring >70
  • Blog posts before they go live
  • Token security scan results (client deliverables)
  • System configuration changes
  • Any task where an error would cost >$100

The ROI

Human review adds ~2 hours/day of CEO time, but prevents costly mistakes. A single mispriced proposal could lose a $5,000 client.

Key insight: Optimize for business value, not just API spend. Free LLMs + strategic human oversight = maximum ROI.

Pattern 7: Rate Limit Budgets + Smart Stops

Even free tiers have limits. Manage them proactively.

Rate Limit Enforcement

FREE_TIER_QUOTAS = {
    "stepfun": {"requests_per_day": 1000, "tokens_per_minute": 10000},
    "claude-free": {"requests_per_day": 100},
    "gpt-4o-mini": {"tokens_per_minute": 20000},
}

# Check before each call
if requests_today[model] > QUOTA_DAILY[model] * 0.8:
    # 80% used → switch to alternative model
    model = fallback_model(model)

When a quota hits 100%, non-critical agents pause until the window resets. Critical revenue tasks trigger alerts for manual intervention.

Result: Predictable throughput. No surprise rate limit blocks.

  • Route by criticality — only use paid models when absolutely necessary

Common Cost-Killing Mistakes (Avoid These)

  • ❌ Using GPT-4 for Everything
    Free tiers exist. Use them. Only upgrade if you've measured a quality gap.
  • ❌ Unbounded Context
    Even with free tiers, sending 10,000 tokens when 500 would do hurts throughput. Set limits.
  • ❌ No Caching
    Recomputing the same thing wastes quota. Cache aggressively.
  • ❌ Real-Time Everything
    Batching maximizes free tier efficiency. Batch what you can.
  • ❌ Ignoring Multi-Provider Strategy
    Don't rely on one free tier. Spread across StepFun, Claude free, GPT-4o-mini to avoid single-point rate limits.
  • ❌ No Rate Limit Budgeting
    Track requests/day and tokens/minute. Know your quotas and stay within them.
