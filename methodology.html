Methodology

How we approach smart contract security research: systematic, evidence-based, and uncompromising on proof.

Core Principles

Working PoC or It's Not a Bug

Theory is worthless. We build proof-of-concept exploits for every finding. If forge test doesn't pass, the vulnerability doesn't exist. No exceptions.

Every submission includes a working exploit, quantified impact in dollar terms, and the exact conditions required to trigger it.

Evidence Over Opinion

We cite real code, real transaction hashes, and real exploit amounts. When we say "$116M lost to oracle manipulation," we link to the Mango Markets exploit transaction on Solscan.

No speculation. No "this could theoretically be exploited." Either we can steal money with working code, or it's secure.

AI-Augmented Research

We use AI as a force multiplier for security research. Pattern recognition, code analysis, and documentation generation happen faster with AI assistance.

But the final decision on every finding requires human judgment. AI helps us analyze faster; it doesn't replace the security researcher.

Research Process

1. Target Selection

We prioritize protocols that match specific criteria:

  • Fresh code: Recent deployments or upgrades where patterns may not be hardened yet
  • Limited audit history: 0-1 prior audits from smaller firms
  • Math complexity: Non-trivial calculation chains where rounding errors or edge cases hide
  • Economic value: Significant TVL or bounty pool to justify research time

2. Understanding

Before hunting vulnerabilities, we map the entire protocol:

  • Trust boundaries: Who can call what, and with what privileges?
  • State transitions: How does user action X affect system state Y?
  • External dependencies: Oracles, cross-contract calls, price feeds
  • Calculation chains: Math operations from input to output, looking for precision loss

3. Kill Zone Identification

We tag high-risk areas for deep analysis:

  • Reentrancy surfaces (external calls before state updates)
  • Access control gaps (missing checks, bypassable modifiers)
  • Math errors (overflow, underflow, rounding to zero)
  • Oracle manipulation (stale data, price impact, TWAP bypass)
  • Flash loan attacks (borrow → manipulate → profit → repay)
  • Economic exploits (arbitrage, MEV, donation attacks)

4. Attack Development

For each kill zone, we attempt exploitation:

  1. Build a Foundry test that reproduces the vulnerable scenario
  2. Develop the attack vector with real attacker constraints (gas costs, capital requirements)
  3. Run the exploit: forge test -vvv must show funds stolen or system broken
  4. Quantify impact: How much can be stolen? Under what conditions?

5. Verification

Before reporting, every finding passes a kill test:

  • Real impact: Can this actually steal or lock funds?
  • Measurable: Quantified dollar amount with real numbers?
  • Reachable: Can a real attacker trigger this with public functions?
  • In scope: Within the bug bounty program rules?
  • Not admin: Doesn't require trusted roles? ("Admin can rug" = invalid)
  • PoC passes: forge test -vvv shows PASS?
  • Economically viable: Profit exceeds gas costs?

Tooling

Static Analysis

We use Slither for automated pattern detection, but we don't trust its output blindly. False positives get cataloged and filtered. Every static analysis alert requires manual verification.

Dynamic Testing

Foundry is our primary testing framework. We fork mainnet state when testing live protocols, isolate test environments for math proofs, and use fuzzing when input space is large.

Manual Review

The majority of vulnerabilities come from reading code and asking "what if?" Static analysis catches known patterns. Manual review catches novel attack vectors.

What We Don't Do

  • Inflate severity: If it's MEDIUM, we call it MEDIUM. Honesty earns trust.
  • Report without PoC: No working exploit = no submission.
  • Guess impact: We quantify every finding with real numbers.
  • Ignore economics: A $100 exploit that costs $500 in gas isn't valid.
  • Rush to submit: We verify findings multiple times before reporting.

Continuous Learning

Every protocol we analyze teaches us something. We document patterns, false positives, and successful attack vectors in an internal knowledge base.

When we're wrong, we document why. When a protocol is secure despite looking vulnerable, we document the defense pattern. This accumulated knowledge makes us faster and more accurate over time.
