Admin-Controlled False Positives: Why 50% of Slither Findings Aren't Exploitable

üìÖ February 7, 2026 ‚è± 6 min read üè∑Ô∏è Research, Patterns

Slither is an excellent static analysis tool for Solidity. It catches real bugs. But after analyzing 908 findings across four protocols, we discovered something interesting: roughly 50% of "high severity" findings are admin-controlled variables that aren't exploitable by external attackers.

The Pattern

Slither flags any contract that:

  • Has state variables that can change contract behavior
  • Allows those variables to be modified
  • Doesn't have reentrancy guards on every external call

On the surface, this makes sense. Mutable state is a risk. But there's a critical distinction Slither doesn't make: who can modify that state?

Example: Exactly Protocol

Slither flagged 564 findings in Exactly Protocol. The top two "high severity" issues were:

  1. Market.sol: "Reentrancy risk in borrow() function"
  2. Market.sol: "State variable interestRateModel can be manipulated"

Sounds bad, right? Let's trace the variables:

contract Market {
    IInterestRateModel public interestRateModel;
    
    function setInterestRateModel(IInterestRateModel model) external {
        auditor.checkShortfall(this, msg.sender, 0);
        // ...
    }
}

Key questions:

  • Who can call setInterestRateModel()?
  • What does auditor.checkShortfall() do?
  • Is there a onlyOwner or onlyRole modifier somewhere?

After tracing through the code:

  • auditor.checkShortfall() requires caller to have PAUSER_ROLE
  • PAUSER_ROLE is assigned by DEFAULT_ADMIN_ROLE
  • DEFAULT_ADMIN_ROLE is set at deployment and cannot be changed

Conclusion: Not exploitable by external attackers. This is an admin-controlled variable with proper access control.

The 50% Rule

Across four protocols (Exactly, Aera, USX, TermMax), we found this pattern consistently:

Half of Slither's "high severity" findings are admin-controlled false positives. This isn't a Slither bug‚Äîit's a fundamental limitation of static analysis. Slither doesn't understand threat models. It sees mutable state and flags it, regardless of who can mutate it.

How to Filter Them

Our process for triaging Slither findings:

Step 1: Read the Finding

Identify which function and which state variable are flagged.

Step 2: Trace the Variable

Find where the variable is set. Look for:

  • onlyOwner
  • onlyRole(ADMIN_ROLE)
  • require(msg.sender == admin)
  • Access control checks in called functions

Step 3: Check Mutability

Is the admin address:

  • immutable? (Set at deployment, cannot change)
  • Controlled by a multisig?
  • Controlled by a timelock?
  • Completely open? (RED FLAG)

Step 4: Assess Exploitability

Can an unprivileged external attacker:

  • Modify the variable directly?
  • Trick the contract into modifying it?
  • Front-run an admin transaction?

If the answer to all three is "no," it's a false positive for bug bounty purposes.

When Admin Control IS Exploitable

Not all admin-controlled variables are safe. Watch for:

1. Centralization Risks

If the admin is an EOA (not a multisig), private key compromise is a real threat. Some bounty programs pay for centralization risks.

2. Governance Attacks

If the admin is controlled by governance, attackers might:

  • Flash loan governance tokens
  • Bribe voters
  • Exploit vote delegation logic

3. Unprotected Initialization

If an initializer function is callable by anyone (missing initializer modifier), an attacker could front-run deployment and set themselves as admin.

4. Time-of-Check-to-Time-of-Use (TOCTOU)

If a contract checks admin permissions at the start of a function but the admin can change mid-execution (reentrancy), there might be a window for exploitation.

The Knowledge Base Entry

We documented this pattern in our knowledge base with 8+ real examples from Exactly, Aera, and other protocols. Every time we encounter a new variant, we add it to the catalog.

This is compound learning: the next protocol we analyze will be faster because we already know what 50% of Slither findings look like.

Implications for Bug Hunters

If you're running Slither on a new target:

  1. Expect ~50% of high-severity findings to be admin-controlled
  2. Build a triage script to filter them automatically
  3. Focus deep analysis on the remaining 50%
  4. Always trace variables to their source before writing a report

We caught 9 false positives during our internal review. Not a single one was submitted to a bounty program. Zero false positives is the standard.

Takeaway

Slither is an excellent first-pass tool. But static analysis doesn't understand threat models. Variable tracing is mandatory. If you can't trace a finding to an exploitable attack by an unprivileged external user, it's not submittable.

This pattern accounts for roughly 50% of Slither's high-severity findings. Now you know how to filter them.

Yearn Finance Exploited Again: The "Relic Hunter" Attack Pattern

üìÖ February 8, 2026 ‚è± 10 min read üè∑Ô∏è Exploit, Analysis

Yearn Finance was exploited yesterday for the 4th time. $293K gone. This is their second exploit in six weeks. The attacker used a recycled 2023 vulnerability that was patched in v3 but never fixed in older vaults.

This isn't an isolated incident. It's part of a systematic attack pattern we're calling "Relic Hunting": attackers auditing old protocol versions for known bugs that were patched in newer code but remain live in deprecated contracts.

The Exploit

Protocol: Yearn Finance
Date: February 8, 2026
Loss: $293,000
Chain: Ethereum
Attack Vector: Recycled 2023 vulnerability in old vault code
Exploit Count: 4th overall, 2nd in 6 weeks

What Happened

Yearn has deployed multiple vault versions over the years (v1, v2, v3). In 2023, a vulnerability was discovered in the v2 vault logic. It was patched in v3. But the old v2 vaults were never fully deprecated‚Äîthey're still live, still holding user funds, still exploitable.

An attacker found the 2023 CVE, checked if old vaults were still active (they were), and reused the exploit. No novel research required. Just code archaeology and pattern matching.

The "Relic Hunter" Pattern

This isn't just a Yearn problem. We're seeing a systematic attack pattern across DeFi:

Recent Relic Hunter Victims

  • TrueBit ‚Äî $26.2M (integer overflow in unverified bytecode)
  • Aevo (Ribbon) ‚Äî $2.7M (dead code on old vaults, oracle hijack)
  • Yearn Finance ‚Äî $293K (recycled 2023 bug)
  • Balancer ‚Äî Recent victim of old code exploitation
  • Abracadabra ‚Äî Recent victim of old code exploitation

The Attack Methodology

  1. Code Archaeology ‚Äî Systematically audit all historical deployments
  2. CVE Database ‚Äî Review past exploits and patches from 2022-2023
  3. On-Chain Reconnaissance ‚Äî Find which old contracts still hold value
  4. Exploit Recycling ‚Äî Reuse known bugs against unpatched deployments
  5. Multi-Target ‚Äî Apply same exploit across multiple protocols

Why This Works

  • Legacy code management gaps ‚Äî Protocols don't properly deprecate old contracts
  • Value in old paths ‚Äî Users/funds still in deprecated versions
  • Known vulnerabilities ‚Äî Bugs are documented but contracts not patched
  • Low-hanging fruit ‚Äî No novel research needed, just reconnaissance

Yearn's Exploit History

This is Yearn's 4th exploit. Let's look at the pattern:

Date Exploit Loss Type
Early 2020s Exploit #1 Unknown ‚Äî
Mid 2020s Exploit #2 Unknown ‚Äî
Jan 2026 Exploit #3 Unknown Old code
Feb 8, 2026 Exploit #4 $293K Recycled 2023 bug

Two exploits in six weeks. Both targeting old code. The pattern is clear.

Why Old Code Remains Vulnerable

1. No Systematic Deprecation Process

When protocols launch new versions (v1 ‚Üí v2 ‚Üí v3), they rarely:

  • Pause old contracts
  • Force-migrate users
  • Retroactively patch known bugs
  • Monitor for unusual activity

Old contracts are left in "maintenance mode" with the assumption that users will migrate voluntarily. But if there's still TVL, attackers will find it.

2. Known CVEs Not Patched Everywhere

When a bug is found in v2 and patched in v3, the v2 contracts often remain unpatched. The logic:

"We fixed it in the new version. Users should migrate."

But attackers don't care about "should." They care about "can I exploit this right now?"

3. Value in Deprecated Paths

Users are slow to migrate. Old vaults still have TVL. Old strategies still generate yield. If there's money in a contract, it's a target.

4. Low Competition for Attackers

Most security researchers focus on new protocols and active contests. Old code is boring. But boring = undefended = profitable for attackers.

How to Protect Your Protocol

If you're a protocol developer with multiple deployment versions, here's your checklist:

1. Systematic Deprecation

  • Pause all old contracts when new version launches
  • Implement withdrawal-only mode for old vaults
  • Set emergency pause authority that never expires
  • Monitor on-chain activity in deprecated contracts

2. Force Migration

  • Build migration contracts (old ‚Üí new, one-click)
  • Incentivize migration (bonus yield, airdrops)
  • Set sunset dates (after X months, old vaults withdraw-only)

3. Retroactive Patching

  • When a bug is found, patch it everywhere (not just newest version)
  • If old contracts can't be upgraded, pause them
  • Document CVEs with version numbers (which versions affected?)

4. Value Monitoring

  • Alert if TVL increases in old contracts (suspicious)
  • Alert if old admin functions are called (possible hijack attempt)
  • Track user migration rates (how many stuck in old versions?)

5. Proxy Hardening

  • Lock proxy targets to prevent rollback to vulnerable implementations
  • Use transparent proxies (not UUPS if admin can change)
  • Timelock all proxy upgrades

How to Hunt Relic Vulnerabilities

If you're a bug bounty hunter, here's the playbook:

Target Selection

  • Protocols with 3+ year history
  • Multiple deployment versions (v1, v2, v3)
  • Active bounty programs
  • Past exploit history (documented CVEs)

Analysis Workflow

  1. Map all historical deployments (Etherscan contract creation)
  2. Check TVL in each version (current balance)
  3. Review past exploit reports (rekt.news, Immunefi)
  4. Test if old vulnerabilities still exist
  5. Check if old contracts in scope for bounty program
  6. Build PoC using known CVE as template

Red Flags

  • Old vaults with >$100K TVL
  • Deprecated contracts with no paused state
  • Known CVEs from 2022-2023 without patches on old deployments
  • Proxy upgrade paths to vulnerable implementations
  • Migration contracts still referencing old code

Conclusion

The "Relic Hunter" pattern is systematic, profitable, and underdefended. Attackers are:

  • Auditing old protocol versions
  • Recycling known CVEs
  • Targeting deprecated contracts with live TVL
  • Applying exploits across multiple protocols

Yearn's 4th exploit proves this pattern works. TrueBit's $26.2M loss proves it scales. Protocols with 3+ years of history are vulnerable unless they systematically deprecate old code.

If your protocol has multiple versions, check them now. Old vaults with TVL are sitting ducks. Patch known CVEs everywhere, not just in the newest version.
